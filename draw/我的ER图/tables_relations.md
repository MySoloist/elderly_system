# 数据库表关系详细分析

## 表关系概览

本系统共有31张表，主要包括用户管理、老人服务、家属服务、配送服务、订餐服务、健康管理等模块。

## 一、用户管理模块(未)

### 1. users表
**表类型**：核心用户表
**关系类型**：
- **一对一关系**：与 `admin_profiles`、`member_profiles`、`elderly_profiles`、`deliverer_profiles` 通过 `user_id` 关联
  **关系名称**：拥有
- **一对多关系**：
  - 与 `ai_conversations` 通过 `user_id` 关联
    **关系名称**：进行
  - 与 `deliverer_locations` 通过 `deliverer_id` 关联
    **关系名称**：记录
  - 与 `deliveries` 通过 `deliverer_id` 关联
    **关系名称**：负责
  - 与 `elder_member_relations` 通过 `elder_id` 和 `member_id` 关联
    **关系名称**：绑定
  - 与 `favorites` 通过 `elderly_id` 关联
    **关系名称**：收藏
  - 与 `favorites_member` 通过 `member_id` 关联
    **关系名称**：收藏
  - 与 `health_records` 通过 `elderly_id` 和 `created_by` 关联
    **关系名称**：记录
  - 与 `health_reminders` 通过 `sender_id` 和 `receiver_id` 关联
    **关系名称**：发送/接收
  - 与 `emergency_calls` 通过 `elderly_id` 关联
    **关系名称**：发起
  - 与 `orders` 通过 `elderly_id` 关联
    **关系名称**：创建
  - 与 `reviews` 通过 `elderly_id` 和 `deliverer_id` 关联
    **关系名称**：评价
  - 与 `staff_schedules` 通过 `staff_id` 关联
    **关系名称**：分配
  - 与 `voice_synthesis` 通过 `user_id` 关联
    **关系名称**：生成

## 总结

本系统的数据库设计遵循了以下原则：

1. **核心用户表设计**：以 `users` 表为核心，通过不同的详情表存储不同类型用户的信息
2. **一对一关系**：用户与用户详情表之间采用一对一关系，确保数据完整性
3. **一对多关系**：大部分业务表之间采用一对多关系，如社区与老人、餐品分类与餐品等
4. **多对多关系**：通过中间表实现多对多关系，如老人与家属的绑定关系、用户与餐品的收藏关系
5. **业务关联性**：表之间的关联紧密反映了实际业务流程，如订单-配送-支付的完整链路

所有表之间的关系设计合理，能够满足系统的业务需求，同时保持了数据的一致性和完整性。
